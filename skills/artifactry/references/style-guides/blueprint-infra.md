# Blueprint Infra

Blueprint Infra is a technical architecture style for system design, infrastructure plans, platform strategy, developer documentation, API maps, and engineering leadership decks. It inherits from infrastructure providers, database systems, documentation platforms, code-forward developer tools, and blueprint-like AI infrastructure brands.

## Core Idea

Make invisible systems visible. The artifact should explain how components connect, where data moves, what depends on what, and which decisions matter.

## Visual Signature

- Blueprint or technical paper canvas.
- Grid lines, node maps, dependency arrows, sequence lanes.
- Mono labels, file paths, API endpoints, and version markers.
- Deep navy, black, white, cyan/blue, green, and amber accents.
- Diagrams that feel functional rather than decorative.

## Palette

Use dark navy or off-white technical canvas. Blue/cyan can indicate active flow, green success/runtime, amber warning/config, red failure/risk. Keep base surfaces quiet so diagrams stand out.

## Typography

Use a sturdy sans for headings and body, mono for all technical identifiers. Diagram labels must be large enough for screenshots. Use uppercase only for component categories or status labels.

## Layout System

Use component diagrams, sequence diagrams, swimlanes, and architecture panels. Keep connection paths simple. If a diagram becomes too dense, split it across multiple slides: context, flow, failure modes, deployment plan.

## Slide Recipes

- Cover: system name, purpose, version, owner.
- Context map: actors, tools, data stores, outputs.
- Request flow: input -> service -> model/tool -> output.
- Deployment: environments, boundaries, release path.
- Dependency map: internal/external services with risk markers.
- Data contract: fields, schema, validation, failure behavior.
- Incident path: detection, triage, mitigation, rollback.
- Closer: architecture decision record summary.

## DOCX Translation

DOCX should behave like an engineering spec. Use diagrams as embedded images, tables for contracts, code blocks for examples, and ADR-style sections: context, decision, consequences.

## Agent Rules

- Ask for missing system boundaries before diagramming.
- Do not invent APIs, services, or schemas.
- Prefer multiple clear diagrams over one overloaded diagram.
- Use arrows only when direction matters.
- Validate code blocks and path labels do not overflow.

## Avoid

Avoid fake technical complexity, random network nodes, unreadable architecture spaghetti, neon cyber visuals, and diagrams without legends.
