# Terminal Operator

Terminal Operator is a dark, command-native style for developer workflows, agent operations, API guides, automation playbooks, incident notes, and technical demos. It inherits from terminal products, IDE surfaces, local model tools, agent frameworks, and code-first infrastructure systems.

## Core Idea

The artifact should feel like a working operating console. It is not a generic dark theme. It must communicate process, state, command, result, and next action.

## Visual Signature

- Near-black canvas with slightly lifted panels.
- Mono labels, command lines, prompt marks, and status chips.
- Green, blue, amber, or violet accents used as state signals.
- Code blocks, trace panels, logs, route diagrams, and terminal windows.
- Compact but readable density with strong left alignment.

## Palette

Use layered dark surfaces: void black, panel black, elevated charcoal, muted text gray, and one or two signal colors. Green means success or online; amber means warning; red means failure; blue/violet can mean active agent or selected path. Do not use random neon everywhere.

## Typography

Mono is a first-class display language, but body text may use a clean sans for readability. Use mono for commands, IDs, filenames, timestamps, model names, API paths, and metrics. Keep code text large enough to read after screenshot export.

## Layout System

Use operational panels: command, output, status, trace, config, queue, result. Prefer stacked windows, split console views, and labeled modules. Margins can be tighter than editorial styles, but still need safe export padding.

## Slide Recipes

- Cover: large command phrase, prompt mark, short result line.
- Workflow: command -> agent action -> output -> validation.
- Trace: numbered log rows with status colors.
- Architecture: service boxes connected by command lines.
- API guide: endpoint, request, response, caveat.
- Incident: timeline, failure, mitigation, current status.
- Checklist: preflight, install, render, validate, deliver.
- Closer: final command and output files.

## DOCX Translation

DOCX should avoid full dark pages unless PDF-only. Translate the style into mono headings, code blocks, command tables, and shaded terminal snippets. Use printable light backgrounds for long documents.

## Agent Rules

- Convert steps into commands and observable outputs.
- Use status labels consistently: pending, running, done, blocked, failed.
- Never make code decorative. It must be meaningful or removed.
- Do not render final text inside screenshots unless the screenshot is the artifact itself.
- Validate contrast on dark surfaces.

## Avoid

Avoid generic cyberpunk glow, hacker stereotypes, unreadable tiny code, random matrix patterns, decorative terminal windows with fake content, and too many simultaneous accent colors.
